# Day 1 — Two Sum (LC #1)

**Problem link:** [leetcode.com/problems/two-sum](https://leetcode.com/problems/two-sum/)

> **Topic this week:** Arrays & Hash Maps. Hash maps are the single most frequently used data structure in competitive programming — mastering them early pays dividends for the next 167 days.

> **Why this problem:** Two Sum is the canonical hash-map problem. It forces you to think about trading memory for time, which is the core trade-off behind almost every O(n) solution that improves on an O(n²) brute force.

## Problem

You are given an array of integers `nums` and a single integer `target`. Return the **indices** (positions in the array, starting from 0) of the exactly two numbers that add up to `target`. The problem guarantees exactly one valid pair exists, and you cannot use the same element twice.

**Examples:**

- Input: `nums = [2, 7, 11, 15]`, `target = 9` → `[0, 1]`  
  (nums[0] + nums[1] = 2 + 7 = 9)
- Input: `nums = [3, 2, 4]`, `target = 6` → `[1, 2]`  
  (nums[1] + nums[2] = 2 + 4 = 6)
- Input: `nums = [3, 3]`, `target = 6` → `[0, 1]`  
  (Same value, different indices — both indices are valid)

**Constraints:**
- 2 ≤ `nums.length` ≤ 10,000
- −10⁹ ≤ `nums[i]` ≤ 10⁹
- −10⁹ ≤ `target` ≤ 10⁹
- Exactly one valid answer exists

## Try it first

Spend 10–20 minutes attempting this before reading the hints. Open a file `solution.cpp` and try a brute-force first — two nested loops checking every pair. Get that working, then ask: can I find the answer in a single pass?

## Hints

<details>
<summary>Hint 1 — pattern recognition</summary>

For each number `x` in the array, you need to find some other number that equals `target - x`. Instead of scanning the whole array again every time, what if you remembered the numbers you've already seen?

</details>

<details>
<summary>Hint 2 — data structure choice</summary>

A hash map lets you answer "have I seen value V before, and if so, at which index?" in O(1) time. If you store `{value → index}` as you walk through the array, you can check for the complement instantly.

> **O(1):** "constant time" — the operation takes the same amount of work regardless of how large the input is. A hash map lookup is O(1) on average.

</details>

<details>
<summary>Hint 3 — the algorithm</summary>

Walk left-to-right through `nums`. At index `i`, compute `complement = target - nums[i]`. Check if `complement` is already in your hash map. If yes, you found your pair — return `{map[complement], i}`. If no, store `nums[i] → i` in the map and continue. Because the problem guarantees one answer, you will always find it before the loop ends.

</details>

## Approach

The key insight is that for any element `x`, its required partner is exactly `target - x`. Call this the **complement**. The brute-force approach checks every pair (i, j) — that's O(n²) time for an array of size n, which becomes 100 million operations on the largest inputs and will be too slow.

The hash-map approach is a classic **look-before-you-leap** technique. Instead of asking "is there a number that completes me?" by scanning the rest of the array, we flip the question: "as I process each element, have I already seen its complement in a previous position?" This requires storing every element we've visited alongside its index — exactly what a hash map does.

Notice that we add the current element to the map **after** checking for its complement. This is important: it prevents us from accidentally pairing an element with itself. For the `[3, 3], target = 6` case, when we reach index 1 (value 3), complement is 3, and we already stored `{3 → 0}` from index 0, so we correctly return `{0, 1}`.

Let's walk through `nums = [2, 7, 11, 15], target = 9`:

| i | nums[i] | complement | seen before? | map after step       |
|---|---------|------------|--------------|----------------------|
| 0 | 2       | 7          | no           | {2 → 0}              |
| 1 | 7       | 2          | **yes (i=0)**| return {0, 1}        |

The answer is found at index 1 after just two iterations — a single pass through the array.

## C++ Solution

```cpp
#include <iostream>
#include <vector>
#include <unordered_map>  // std::unordered_map: a hash map (key→value lookup in O(1) avg); not yet covered in cpp-quant — think of it as a dictionary

class Solution {
public:
    std::vector<int> twoSum(std::vector<int>& nums, int target) {
        // std::vector<int>: a resizable array; covered on cpp-quant Day 1 foundations
        std::unordered_map<int, int> seen; // maps value → index

        for (int i = 0; i < (int)nums.size(); ++i) {
            int complement = target - nums[i];

            // .count(key) returns 1 if the key exists, 0 if not
            if (seen.count(complement)) {
                return {seen[complement], i}; // found the pair
            }

            seen[nums[i]] = i; // store current element AFTER checking
        }

        return {}; // unreachable — problem guarantees exactly one answer
    }
};

int main() {
    Solution sol;

    std::vector<int> nums1 = {2, 7, 11, 15};
    auto r1 = sol.twoSum(nums1, 9);
    std::cout << "[" << r1[0] << ", " << r1[1] << "]\n"; // expected: [0, 1]

    std::vector<int> nums2 = {3, 2, 4};
    auto r2 = sol.twoSum(nums2, 6);
    std::cout << "[" << r2[0] << ", " << r2[1] << "]\n"; // expected: [1, 2]

    std::vector<int> nums3 = {3, 3};
    auto r3 = sol.twoSum(nums3, 6);
    std::cout << "[" << r3[0] << ", " << r3[1] << "]\n"; // expected: [0, 1]

    return 0;
}
```

**Compile and run locally:**

```bash
g++ -std=c++17 -O2 -Wall solution.cpp -o solution
./solution
```

```
[0, 1]
[1, 2]
[0, 1]
```

## Complexity

- **Time:** O(n) — we visit each element once, and each hash map lookup/insert is O(1) on average.
- **Space:** O(n) — in the worst case (no complement found until the last step) we store every element in the hash map.

## Common Pitfalls

- **Storing before checking.** If you do `seen[nums[i]] = i` before the `count(complement)` check, an element could match itself. For `nums = [6], target = 12`, this would incorrectly return `{0, 0}`. Always check first, then store.
- **Returning sorted indices vs. original order.** The problem asks for indices in the original array. Don't sort the array to simplify the search — you'd scramble the indices. The hash-map approach avoids this entirely.
- **Assuming non-negative values.** The constraints allow negative numbers and a negative target. The complement arithmetic `target - nums[i]` works correctly for all integers, so there's nothing special to handle.
- **Using `int` carelessly.** The constraints say |nums[i]| ≤ 10⁹ and |target| ≤ 10⁹. The complement `target - nums[i]` can reach 2 × 10⁹ in magnitude, which overflows a 32-bit `int` (max ~2.1 × 10⁹). Here it's fine because `target` and `nums[i]` are both `int` and their difference fits, but on similar problems be alert to overflow — use `long long` when sums could exceed 2 × 10⁹.
- **Off-by-one on the loop bound.** `nums.size()` returns an unsigned type (`size_t`). Comparing a signed `int i` against an unsigned value can produce surprising results when the array is empty (size 0 would underflow). The cast `(int)nums.size()` or using `i < (int)nums.size()` is the safe pattern.

## Quant Relevance

Symbol-to-price lookups in a market data feed are a direct application of this pattern: given a ticker symbol (key), retrieve the last price (value) in O(1). Order routing engines use hash maps the same way — instrument ID → order book reference — to handle thousands of lookups per second without iteration.
